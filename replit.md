# FrontDesk — Studio Guest Management System

A web-based guest management system for broadcast studio security desks — check guests in/out, manage pre-registrations, watchlists, and audit logs. Branded as **FrontDesk** (OBTV Studio Security). Brand assets live in `artifacts/studio-gms/public/` (favicon.ico/.svg, `frontdesk-icon-*.png` including the 180px apple-touch icon, `frontdesk-logo-*.svg`); the in-app mark is `public/logo.svg`. The mark is a security counter with a signal-teal presence dot (= on-site status).

## Run & Operate

- `pnpm --filter @workspace/api-server run dev` — run the API server (port 8080)
- `pnpm --filter @workspace/studio-gms run dev` — run the frontend (port 19603)
- `pnpm run typecheck` — full typecheck across all packages
- `pnpm run build` — typecheck + build all packages
- `pnpm --filter @workspace/api-spec run codegen` — regenerate API hooks and Zod schemas from the OpenAPI spec
- `pnpm --filter @workspace/db run push` — push DB schema changes (dev only)
- Required env: `DATABASE_URL` — Postgres connection string (auto-provisioned)
- Required env: `SESSION_SECRET` — secret used to sign session cookies (must be set; server refuses to start without it)
- Optional env: `BOOKINGS_API_URL` — base URL of the external bookings API for the dashboard "Productions Today" panel (defaults to `https://plex.bookstud.io/api/public/bookings`). The server appends `start`/`end` query params for today's UTC window and proxies the result via `GET /api/productions/today`.
- Optional env: `MAXXESS_BRIDGE_TOKEN` — shared secret for the Maxxess access-control bridge (`bridge/maxxess-bridge/`). Unset = occupancy ingest disabled (`POST /api/integrations/occupancy` returns 503). Also exposed in docker-compose (`${MAXXESS_BRIDGE_TOKEN:-}`).
- Optional env: `APP_BASE_URL` — public base URL used to build one-click approval links in approver emails (e.g. `https://sec.obtv.io`). Link-base precedence: `APP_BASE_URL` → `REPLIT_DEV_DOMAIN` (dev) → **learned public origin** — the server records the request origin of security/admin-authenticated requests only (`lib/public-origin.ts`, hooked in `requireOperator`/`requireAdmin` AFTER the role check; kiosk/client/unauthenticated requests must NEVER teach the origin — host-header injection would poison approval links and exfiltrate decision tokens; this was an architect-flagged vuln, do not "simplify" back to per-request origin). Learned origin persists in `app_settings.app_public_origin`, loaded at startup. Until a staff sign-in happens, emails omit the buttons and direct approvers to the in-app Approvals page. `deploy/nginx.conf` preserves an outer proxy's `X-Forwarded-Proto` (http-level `map`) so origins learn `https://` behind TLS. Approver emails contain separate green **Approve** / red **Deny** buttons linking to `/approval/:token?action=approve|deny`; the public decision page highlights the intended action and still requires a confirming button press (mail-scanner prefetch can never decide).
- Optional env: `SENDGRID_API_KEY` (secret) + `SENDGRID_FROM_EMAIL` — enable outbound email alerts for visitor events. Both must be set (uses `@sendgrid/mail` directly, no Replit connector, so it works in self-hosted Docker). If either is missing the server logs a warning at startup and email alerts are a silent no-op; everything else works. `SENDGRID_FROM_EMAIL` must be a verified SendGrid sender/domain. Check live status via `GET /api/alert-status`.
- Frontend env: `VITE_SITE_NAME` (in `artifacts/studio-gms/.env`) — the single static site/location name shown across the app. There is no in-app site selector; change this env var + rebuild to deploy for a different location. Must restart the `web` workflow after changing it.
- Frontend env (client branding for the PUBLIC `/preregister` page): `VITE_CLIENT_NAME` (shown under the logo; blank falls back to `VITE_SITE_NAME`) and `VITE_CLIENT_LOGO` (a filename in `artifacts/studio-gms/public/` such as `client-logo.png`, or a full URL; blank falls back to the default FrontDesk mark). Resolved in `src/lib/site.ts` as `CLIENT_NAME` / `CLIENT_LOGO_URL`. To rebrand for a new client: drop their logo in `public/`, update these two env vars, and restart the `web` workflow.
- Seed the fixed admin: `node scripts/src/seed-admin.mjs` (defaults to `admin@studiogms.com` / `StudioAdmin!2026`; override with `SEED_ADMIN_EMAIL` / `SEED_ADMIN_PASSWORD`)

## Deployment target

- **Self-hosted**: must run as a Docker container on the user's own Ubuntu server (not Replit Deployments). Keep everything container-friendly: bind to `PORT`, no Replit-only runtime assumptions, all config via env vars. Auth is fully self-contained (username/password + Postgres-backed sessions) — no outbound internet or third-party auth service required. Only `DATABASE_URL` and `SESSION_SECRET` are needed to run; `SENDGRID_API_KEY`/`SENDGRID_FROM_EMAIL` are optional and only needed for email alerts (SendGrid is the sole outbound dependency and is used directly, not via any Replit connector).
- **Docker deploy files** (repo root): `Dockerfile.api` (multi-stage; `builder` runs `pnpm install` + esbuild bundle, `runtime` copies only `dist/` and runs `node artifacts/api-server/dist/index.mjs`), `Dockerfile.web` (`builder` runs the Vite build with `VITE_*` build args baked into `.env`; `runtime` is nginx serving `dist/public` and reverse-proxying `/api` → the api service via `deploy/nginx.conf`), `docker-compose.yml` (services: `db` postgres:16-alpine, one-shot `migrate` reusing the api builder to run drizzle push + seed-admin, `api`, `web`; named volumes `db-data` + `uploads`), `.dockerignore`, `.env.example`, and `DEPLOY.md` (step-by-step Ubuntu instructions). Compose start order: `db` healthy → `migrate` completed → `api` healthy → `web`. `SEED_ADMIN_PASSWORD` is a REQUIRED compose var (no default) so an install can never ship with a known password. `POSTGRES_PASSWORD` must be URL-safe (it's interpolated into `DATABASE_URL`).
- Optional env: `TZ` — IANA timezone (e.g. `America/Chicago`) for the api container. All timestamps are stored UTC and the web UI renders in each viewer browser timezone; TZ only affects server-generated times — alert/approval email timestamps (formatted via toLocaleString with timeZoneName) and the "today" windows (kiosk prereg search, Productions Today, client portal Today's Visits), which use server-local day boundaries. Defaults to UTC.
- Optional env: `SESSION_COOKIE_SECURE` — controls the session cookie `secure` flag (default `false`). Leave `false` for plain HTTP on a LAN/IP; set `true` when serving over HTTPS (e.g. behind a TLS-terminating reverse proxy), otherwise the browser won't send the Secure cookie over HTTP and login silently fails. This replaces the old `NODE_ENV==="production"` check so the default self-hosted HTTP deploy works out of the box.

## Architecture decisions

- Contract-first: OpenAPI spec → codegen → React Query hooks + Zod validators. Never hand-write types that codegen produces.
- Auth is self-contained: `POST /api/auth/login` verifies the bcrypt hash and establishes an httpOnly session cookie (`sid`); `POST /api/auth/logout` destroys it. `requireAuth` middleware checks `req.session.userId`; `requireAdmin` additionally checks role in `app_users` table. Roles: `security` (desk ops) → `supervisor` ("Security Supervisor" — all security access plus watchlist + audit log via `requireSupervisor` on those routes; sidebar shows a "Supervision" section) → `admin` (everything incl. users + app settings); plus `kiosk` and `client`. `requireOperator`/`requireKioskAccess` include supervisor; supervisors are eligible approval approvers. The session store table (`session`) is managed via the Drizzle schema (`createTableIfMissing: false`) so it survives the esbuild Docker bundle.
- `app_users.clerk_id` is the primary key, repurposed as an opaque internal user id (kept its name to avoid contract/codegen churn); new users get a server-generated `usr_<uuid>`. `password_hash` is nullable — users without a hash cannot log in. Email uniqueness is enforced case-insensitively by a DB unique index on `lower(email)` (Postgres error 23505 → 409); the create handler also pre-checks and maps the violation.
- Photo uploads stored on disk under `artifacts/api-server/uploads/` as base64-decoded JPEGs. Served via `/api/photos/:filename`.
- Badge IDs are generated server-side as `GMS-XXXXXX` hex strings on check-in.
- Audit log is append-only; every check-in, checkout, pre-registration, watchlist change, and role change is recorded.

## Product

- **Guest Check-In**: Form with name, company, contact info, host, purpose, studios (checkboxes), expected departure. Site is fixed per deployment. Optional webcam photo. Auto-generates badge. Live watchlist check on name entry — blocks entry for blocked guests, warns for flagged guests. On success, shows a printable landscape visitor badge (photo/initials, name, company, host, purpose, studios, badge ID, in/out times) via the shared `VisitorBadge` component; "Print Badge" triggers `printBadge()`. **Badge/label size is per-workstation**: each security desk sets its own label size in-app via the `BadgeSizeControl` (the "Label size" control next to Print Badge on check-in/prereg-convert and on the dashboard header), stored in that browser's `localStorage` (`frontdesk.badgeSize`, `lib/badge-size.ts`) — because different desks at the same site can have different label printers. `VisitorBadge` reads the size via `useBadgeSize()` and applies inline `width/height` + `data-badge-*` attrs; `printBadge()` reads those attrs to inject a matching `@page { size: W H }` so the badge fills the whole label. Env vars `VITE_BADGE_WIDTH`/`VITE_BADGE_HEIGHT` (`lib/site.ts`, default `3in`/`2in`) only seed the default for a fresh browser. Sizes accept a CSS length in in/mm/cm (validated by `isValidBadgeLength`, safe to interpolate into the injected @page CSS).
- **Guest Check-Out**: Quick search by name or badge ID, one-click checkout with auto-timestamp.
- **Active Dashboard**: Live table of on-site guests with overdue highlights, stats bar (active/today counts/overdue/expected), auto-refresh every 30s. Filter by host/company/site. **Needs Badge**: kiosk self-check-in guests (`checkinSource==='kiosk'` with `badgePrintedAt==null`) are highlighted (amber row + "NEEDS BADGE" pill), sorted to the top, and counted in a header pill ("N badges to print"), so the desk officer can print-and-hand-off at a glance. Clicking **Print Badge** for such a guest also calls `POST /guests/{id}/print-badge` (stamps `badgePrintedAt=now`), which clears the indicator on the next refresh; desk-source check-ins print normally without the indicator. Also shows a **Productions Today** panel — studio bookings scheduled for today, fetched from the external bookings API and proxied via `GET /api/productions/today` (server filters to bookings overlapping today's UTC window, sorted by start time). Configurable via `BOOKINGS_API_URL`.
- **Pre-Registration**: Hosts pre-register expected guests; security sees "Expected Today" queue. Every prereg stores `createdByName` (staff/portal creates store the operator's name; public form stores the visitor-supplied `registeredBy`); the staff `/preregistrations` table shows a "Registered By" column falling back to "Staff"/"Self-registration" for older rows. "Check In" opens a convert dialog with optional webcam photo capture, then converts the pre-reg to an active check-in (`POST /preregistrations/{id}/convert` accepts an optional `{ photoUrl }`) and shows the same printable `VisitorBadge` with a Print button. If the guest is a returning visitor (exact case-insensitive name match in Known Guests), the convert dialog prefills their known-guest photo and passes it through on convert unless the operator clears/retakes it.
- **Public Pre-Registration**: Unauthenticated `/preregister` page — a 4-step wizard (Visitor → Host → Schedule → Review; framer-motion transitions, dark glass styling, per-step validation). Step 1 asks the form-filler's own name FIRST (required `registeredBy`, stored to `createdByName`) with an "I'm the visitor" shortcut that copies it into the visitor name. Submissions land in the same "Expected Today" queue with `createdByClerkId=null`. The success screen shows a prominent amber "Late registration" warning (additional screening notice) when the API returns `lateRegistration=true`. No Layout/auth wrapper. Client-brandable via `VITE_CLIENT_NAME` / `VITE_CLIENT_LOGO` (logo + label on both the form header and the success screen) so the visitor-facing page carries the client's brand, not FrontDesk's.
- **Studios**: Admin-managed list of rooms/buildings (`/studios` page). Multi-selectable via checkboxes on check-in and both pre-reg forms (internal + public). Stored as a `text[]` on guests and preregistrations; copied through on convert. `GET /studios` is public (needed by the public form); `POST`/`DELETE` are admin-only. Displayed in dashboard + prereg tables (replacing the now-static Site column).
- **Email Alerts**: Admin-configurable email notifications for four visitor events — pre-registration created, check-in, check-out, and overdue. Admins manage recipient addresses per event type on the `/alerts` page (admin-only nav). Alerts send via SendGrid (`SENDGRID_API_KEY` + `SENDGRID_FROM_EMAIL`); when unconfigured the page shows a "not configured" banner and no emails are sent. Event triggers are fire-and-forget (`void sendVisitorAlert(...)`) so they never block or fail a check-in/out. Overdue alerts are emitted by a 60s background scheduler (`src/lib/overdue-scheduler.ts`) that finds active guests past `expectedDeparture` with no `overdueAlertSentAt`, and only stamps `overdueAlertSentAt` **after** a confirmed send so a transient email failure retries on the next tick (one overdue email per guest). Endpoints: `GET/POST /alert-recipients`, `DELETE /alert-recipients/{id}`, `GET /alert-status` (all admin-only).
- **Known Guests**: Automatic returning-visitor directory (`/known-guests` page, any operator). Every check-in (direct or pre-reg convert) atomically upserts a `known_guests` profile keyed case-insensitively on name (`INSERT ... ON CONFLICT (lower(name)) DO UPDATE`, new non-null fields win, fire-and-forget so it never blocks check-in — `src/lib/known-guests.ts`). Page shows search, VIP filter, visit count + last visit (LEFT JOIN + GROUP BY over guests by lower(name)), per-guest visit-history dialog, star toggle to mark VIP (audited as `known_guest_vip`), an **Edit** dialog (name/company/phone/email), a **Delete** confirmation (removes the profile only; audit history is untouched and a new profile is recreated on next check-in), server-side **pagination** (20/page, Prev/Next + "showing X–Y of N", page auto-resets to 1 when search/VIP filter changes), and a quick "Check In" that pre-fills the check-in form via sessionStorage `checkin-prefill`. The check-in form's name field shows a typeahead of known guests (≥2 chars) that pre-fills company/phone/email/photo; VIPs show a gold star. Endpoints: `GET /known-guests` (q, vip, page, pageSize → returns `{items,total,page,pageSize}`), `PATCH /known-guests/{id}` (partial: name/company/phone/email/isVip; 409 on lower(name) collision, audited `known_guest_vip` for VIP-only change else `known_guest_edited`), `DELETE /known-guests/{id}` (204/404, audited `known_guest_deleted`), `GET /known-guests/{id}/visits` — all requireAuth (any operator). NOTE: the paginated list response shape means every consumer must read `.items` (both `known-guests.tsx` and the `checkin.tsx` typeahead). Any new `known_guest_*` audit eventType MUST be added to the `AuditEntry.eventType` enum in `openapi.yaml` (the audit-log response validates strictly against it), or `GET /api/audit` 500s once such a row exists.
- **Visit Log**: Operator-accessible (`/visits` page, any operator — not admin-only like the Audit Log) chronological log of ALL visits (check-ins and check-outs). Search by name/badge/company, status filter (on site/checked out/all), from/to date range (inclusive), server-side pagination (25/page, `{items,total,page,pageSize}` shape), 30s auto-refresh. Sortable columns (Guest/Company/Host/Badge/Checked In/Checked Out) via `sortBy`/`sortDir` query params — server-side whitelist column map, `NULLS LAST`, `checkinAt desc` tiebreaker; default sort is Checked In desc. Clicking a guest's name sets the search to that name = quick per-guest history (per-guest history dialogs also exist on Known Guests). Endpoint: `GET /guests/history` (requireOperator; registered before `/guests/:id`). Date params are `YYYY-MM-DD` strings validated by regex in the spec AND semantically in the handler (400 on e.g. `2026-99-99`), compared against `checkinAt` as real Date bounds (`to` is inclusive via < to+1day).
- **Pre-Registration Approvals**: Optional sequential approval workflow for ALL prereg sources (staff, public form, client portal bulk). Admin configures up to two approvers (existing security|admin users) on the `/approvals` page ("Approvals" nav, any operator sees their own queue; config card is admin-only), stored in `app_settings` keys `approval_approver1_id`/`approval_approver2_id`. No approver 1 = workflow disabled = every prereg auto-approved (bypass). With approvers: new preregs get `approvalStatus='pending'`, `approvalStage=1`; approver 2 is only notified AFTER approver 1 approves (sequential). Decisions via (a) the in-app Approvals queue (`GET /approvals/pending` returns ALL pending preregs to any operator — read-only visibility — each annotated with `canDecide` (true only for the current-stage approver) + `awaitingApproverName`; the UI shows Approve/Deny buttons only when `canDecide`, else a "Waiting on <name>" label; `POST /approvals/{id}/decide` still 403s unless caller is the current-stage approver) or (b) one-click email links to the PUBLIC page `/approval/:token` (per-stage unguessable uuid tokens; `GET /approvals/token/{token}` + POST `.../decide` — page requires a button press so mail-scanner prefetch can't decide; idempotent/replay-safe, concurrency-guarded via conditional UPDATE in `lib/approvals.ts` `applyDecision`). Registrations created <4h before arrival are accepted but flagged `lateRegistration=true` (amber LATE pills everywhere, warning banner on token page). Gating: convert → 409 while pending/denied (UI disables Check In with pills on `/preregistrations`), kiosk search/checkin and dashboard Expected Today are approved-only (kiosk shows generic "see security desk"). Requester notified of final outcome (client `notifyEmail??email`, else prereg email). The Approvals page also shows a **Recently Denied** section — `GET /approvals/denied` (requireOperator) returns up to 50 denied-but-still-pending preregs (approvalStatus=denied, status=pending, newest decision first) annotated with `deniedByName`/`deniedAt` (DeniedApproval schema). Audited as `prereg_approved`/`prereg_denied`/`approval_workflow_updated`. Approval emails need SendGrid configured; `APP_BASE_URL` env (optional) sets the link base — falls back to `REPLIT_DEV_DOMAIN`, else emails say "sign in to the Approvals page". Server logic in `artifacts/api-server/src/lib/approvals.ts` + `routes/approvals.ts`; frontend `pages/approvals.tsx` + public `pages/approval-decision.tsx`.
- **Watchlist**: Admin-managed blocklist/flaglist. Name-match check on every check-in. Entries can carry an optional **photo** (`watchlist.photo_url`, captured via the shared `PhotoCapture` in the add dialog, uploaded through `POST /photos`): shown as thumbnails in the watchlist table and, critically, in the check-in form's live watchlist warning/block banner so the officer can visually confirm the match is the same person.
- **Emergency Evacuation Roster**: `/evacuation` page (any operator; red "Emergency" nav section) — full-screen roster of everyone currently on site (name/company/host/studios/badge/check-in time), 15s auto-refresh with a live "as of" timestamp, big print button (print CSS strips the app chrome for a clean paper muster list). Uses the existing active-guests endpoint; no new API.
- **Host Arrival Notifications**: Optional per-visit `hostEmail` on check-in, pre-registration (staff + public forms), and stored on both `guests` and `preregistrations` (copied through on convert; also honored on kiosk self-check-in). When present, check-in fires a fire-and-forget "your guest has arrived" email to the host (`sendHostArrivalNotification` in `lib/alerts.ts`, SendGrid; silent no-op when unconfigured). Independent of the admin-configured alert recipients. Client-portal bulk prereg intentionally has no hostEmail field. An optional **hostPhone** (record-only, no notification) is also captured on all pre-reg forms (staff, public, portal bulk) and direct check-in, stored on both tables and copied through on convert/kiosk convert.
- **QR Fast-Track Check-In**: Every pre-registration gets a unique `fastTrackCode` (`FT-` + 8 hex chars, `generateFastTrackCode` in `lib/badge.ts`; DB unique index `preregistrations_fast_track_code_unique` + collision-retry inserts — ALL prereg inserts must go through `insertPreregWithFastTrackCode`/`insertPreregsWithFastTrackCodes` in `lib/fast-track.ts`, which regenerate codes on 23505). Once approved (or immediately when the approval workflow is off), the guest is emailed their QR code (`sendFastTrackEmail` — QR PNG generated server-side via the `qrcode` package, embedded as an inline `cid` attachment; SendGrid requires snake_case `content_id`; for approval-workflow preregs this email doubles as the approval notice, while client-portal requesters still get the plain outcome notice). At the desk, the "Fast-Track" button on `/preregistrations` opens `FastTrackDialog`: autofocused input (USB barcode scanners type + Enter) plus an optional camera scan via the native `BarcodeDetector` where available; a valid code jumps straight into the existing convert dialog. Endpoint: `GET /preregistrations/fasttrack/{code}` (requireOperator, case-insensitive — uppercases input; 404 unknown/already-converted, 409 approval pending/denied with distinct messages). Codes die on convert (lookup filters `status='pending'`).
- **Building Occupancy (Maxxess bridge)**: Outbound-only LAN bridge (`bridge/maxxess-bridge/` — Node 18+ ESM poller, Dockerfile, README, SQL-SETUP.md; `SOURCE=mock` test data, `SOURCE=efusion-sql` WORKING adapter that reads the eFusion SQL Server DB directly via `mssql` — site: server CONTEGO3, instance MAXXESS, DB `AXxess`, read-only login `frontdesk_reader`; occupants from `CardholderLocation` view (LastAreaName set = inside), events from `CardholderTransactions_V` (Id → externalId `ax-<Id>`); query overrides via `EFUSION_SQL_OCCUPANTS_QUERY`/`EFUSION_SQL_EVENTS_QUERY`; or `SOURCE=efusion` Web API template — NOT licensed at this site) reads Maxxess eFusion occupancy + door events and POSTs them to `POST /api/integrations/occupancy` (Bearer `MAXXESS_BRIDGE_TOKEN`, timingSafeEqual, 503 when env unset — the access system is never internet-exposed). Ingest is transactional: `building_occupancy` snapshot is full-replaced (delete+insert), `access_events` deduped on unique `externalId` (`onConflictDoNothing` + `returning` → accurate `eventsAdded`), 72h event retention prune, last-sync stamped in `app_settings.occupancy_last_sync_at`. Zod caps: 5000 occupants / 1000 events per push (fits the 10mb JSON limit). Reads are operator-only: `GET /occupancy` (occupants + `lastSyncAt`), `GET /occupancy/events?limit=1–200`. Frontend: `/building` page (nav "Building") — searchable occupant table + Recent Door Activity feed, 30s auto-refresh, stale banner when lastSyncAt >3min, "never synced" setup hint; `/evacuation` gains a print-friendly "Staff & Cardholders" section (hidden until first sync, own stale warning, accounted-for checkboxes) and the header counts visitors + cardholders. Schema `lib/db/src/schema/occupancy.ts`; routes `artifacts/api-server/src/routes/occupancy.ts`.
- **ID Scan via paired phone**: "Scan ID" button on the check-in form opens a dialog (`components/scan-id-dialog.tsx`) that creates a 5-min scan session and shows a QR code to `/scan/<sessionId>`; polls every 2s and on completion auto-fills the name (through the watchlist-checking name handler) + photo, then closes. The transferred photo is visible in the form's Photo Capture panel — `PhotoCapture` accepts an optional `photoUrl` prop (already-uploaded URL, e.g. phone-scan transfer or known-guest prefill) shown when no local capture exists; clear/retake clears it. The public phone page (`pages/scan.tsx`, no auth/Layout) reads the **PDF417 barcode on the back of a US driver's license fully on-device** via `zxing-wasm` (lazy import; the `.wasm` binary is bundled locally via `prepareZXingModule` + Vite `?url` import — NOT the default jsDelivr CDN — so it works offline/self-hosted; a warm-up decode surfaces module failures as a visible error). Capture: camera requested at ideal 4K + continuous-focus constraint (PDF417 needs pixel density — 1080p full-frame reliably fails); decode prefers the browser-native BarcodeDetector when it supports pdf417 (Android Chrome — hardware-backed), fed the RAW full-res video element (never a downscaled canvas), and falls back to zxing-wasm (iOS); if the native detector goes ~2s with zero detections (`nativeMisses>=8`), zxing is interleaved every other frame as a second opinion (some devices nominally support pdf417 but never detect dense license barcodes); scanner error only if BOTH are unavailable or persistently failing (10 consecutive exceptions). zxing decode input is bounded (crop pass ≤3200px — no downscale of a 4K guide-box crop, full-frame pass ≤1920px long edge, tryRotate+tryInvert on), alternating center guide-box crop (80%×60%) with downscaled full frame. When the camera track supports zoom (`getCapabilities().zoom.max>1`), 1×/2×(/3×) pill buttons overlay the viewfinder (`button-zoom-N` testids) — zooming in lets the guest hold the license at a focusable distance while keeping pixel density (the dominant real-world failure is focus distance); the scanning UI shows the active camera resolution (data-testid text-cam-res) for remote diagnosis. **Still-photo fallback**: after 5s without a live detection (or immediately when camera/scanner unavailable), a "Take a photo of the barcode" button (`button-still-photo`) opens the NATIVE camera app via `<input type=file capture=environment>` — native stills get real autofocus/macro that getUserMedia video cannot, so a deliberate photo is the most reliable decode path; the still is decoded on-device (native BarcodeDetector, else a zxing ladder: raw File first — wasm decodes the JPEG at FULL native resolution, no canvas downscale — then canvas passes: full frame @3200, middle horizontal band full-width×central-60% @3600, full frame @2000, all tryHarder/rotate/invert/downscale; createImageBitmap with `<img>` fallback), the live loop idles while a still decode runs (stillBusyRef) or the page is hidden, and failures show a retry hint (`text-still-failed`) that distinguishes "barcode decoded but no AAMVA name — maybe not a US license, use manual entry" from "couldn't decode at all". zxing loading is a module-level cached `loadZxing()` shared by both paths. Badge photo capped at 1280px long edge. Camera requires a secure context: over plain HTTP the page shows an explicit HTTPS explanation and falls back to manual name entry (see DEPLOY.md §4), parses ONLY the name from AAMVA data (`parseAamvaName`: DCS/DAC/DAD, legacy DAA fallback, middle "NONE" filtered, title-cased) — the raw barcode/license number never leaves the phone. Flow: scan (or manual-entry fallback when no camera) → editable confirm → live guest photo (rear camera, skippable) → submit. Server (`routes/scan-sessions.ts`): **in-memory** Map w/ TTL sweep (sessions lost on API restart by design; dialog detects poll 404 and offers a new code); `POST /scan-sessions` + `GET /scan-sessions/{id}` + `DELETE /scan-sessions/{id}` (idempotent cancel, 204) are requireOperator; the dialog invalidates the QR token via that DELETE (keepalive fetch) on dialog close, SPA unmount, and pagehide/beforeunload (refresh/tab close), and a generation counter cancels sessions whose create resolved after close — an abandoned QR never lives out its 5-min TTL; the PHONE page also enforces this: unauthenticated `GET /scan-sessions/{id}/status` (204 live+pending, 404 unknown/expired/cancelled/completed; own per-IP rate-limit bucket) is checked on load AND on visibilitychange/pageshow (reload, back-nav, return from native camera app) — a dead session stops the camera and shows a full-screen "link no longer valid" screen (`text-session-dead`), gating the live loop, photo step, and still-photo offer (only a definitive 404 kills the page; transient errors keep state; suppressed on the post-submit done screen since completing the scan makes status 404); `POST /scan-sessions/{id}/submit` is deliberately UNauthenticated — token-authenticated by the unguessable uuid id, single-use (409 on resubmit), per-IP rate-limited (30/min → 429). Photo saved via existing `savePhoto`. QR URL built as `origin + BASE_URL + scan/<id>` (the QR wrapper carries `data-scan-url` for e2e tests). **Scan debugging/telemetry**: the phone posts live scanner telemetry (`ScanDiagnostics`: stage, decoder in use, camRes, zoom, frame/zxing/error/still counters, secureContext, userAgent, lastEvent) every 3s to unauthenticated `POST /scan-sessions/{id}/diagnostics` (own "diag" rate bucket, 20/min < 30/min limit; server stamps `receivedAt`, stored in-memory on the session entry, gone with the session). The desk dialog's 2s poll (`GET /scan-sessions/{id}` now includes optional `diagnostics`) renders a monospace "phone status" block (`scan-phone-diagnostics` testid) — stage/decoder/cam/counters/last event, red warning when the phone reports an insecure (HTTP) context — so the officer can see why a scan is failing without touching the guest's phone. On the phone, a local debug overlay (`scan-debug-panel` testid) shows the same live counters: open with `?debug=1` or by tapping the "ID Scan" header 5×. Notable events logged include camera start/fail (with DOMException name), insecure context, zxing load failure, native-detector fallback, decode errors, "barcode read but no AAMVA name", and each still-photo attempt/outcome.

## Gotchas

- Always run `pnpm --filter @workspace/api-spec run codegen` after changing `lib/api-spec/openapi.yaml` before writing route handlers or frontend hooks.
- `pnpm --filter @workspace/db run push` may need `push-force` if there are column conflicts.
- `express-session` (with `connect-pg-simple`) must be mounted BEFORE the routes in app.ts so `req.session` is populated; the session store reuses the shared pg `pool`.
- `requireAdmin` makes a DB query; don't call it in hot paths.
- Photo uploads have a 10MB body limit set in express.json({ limit: '10mb' }).
- `artifacts/api-server/uploads/` holds runtime visitor photos (PII) and is gitignored — never commit its contents. The dir is auto-created at runtime by `src/lib/badge.ts`.
- Badge printing: the badge card carries `id="print-badge"`; `printBadge()` (in `src/lib/print-badge.ts`) clones it into a `#print-root` child of `<body>` and toggles `body.printing-badge`, which the `@media print` block in `index.css` isolates and sizes to 3in × 2in. Do NOT rely on plain `window.print()` for badges inside a Radix dialog — the portal/transform breaks positioning.
